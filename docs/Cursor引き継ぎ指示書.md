# Cursor引き継ぎ指示書

作成日: 2026-06-24  
対象: ダンプ・現場マッチングアプリ MVP本番開発

## 1. Cursorに渡す資料

Cursorには、以下の順番で資料を渡してください。

| 優先 | 資料 | 用途 |
|---:|---|---|
| 1 | 要件定義書_MVP.md | 何を作るかの中心資料 |
| 2 | DB設計書_MVP.md | データベース、API、権限設計の資料 |
| 3 | 本番化設計メモ.md | 業務ルール、画面一覧、予約ルールの全体像 |
| 4 | `docs/prototype/` フォルダ | 画面デザインと操作感の参考 |

重要:

- `要件定義書_MVP.md` を最優先の仕様として扱う。
- `DB設計書_MVP.md` はSupabase/PostgreSQL実装の土台として使う。
- `本番化設計メモ.md` は業務理解と補足ルールとして使う。
- `docs/prototype/` は試作品なので、そのまま本番コードとして使わず、UI参考にする。

## 2. Cursorに最初に出す指示

以下をそのままCursorに貼ってください。

```text
あなたはWebアプリ開発の上級エンジニアです。
これから「ダンプ・現場マッチングアプリ」のMVP本番版を作ります。

まず、添付した以下の資料をすべて読んでください。

1. 要件定義書_MVP.md
2. DB設計書_MVP.md
3. 本番化設計メモ.md
4. docs/prototype/ フォルダ内の試作画面

優先順位は、要件定義書_MVP.md、DB設計書_MVP.md、本番化設計メモ.md、試作画面の順です。

最初の回答では、まだコードを書かずに、以下だけを出してください。

- 実装方針
- 技術構成案
- ディレクトリ構成案
- MVPで最初に作る順番
- 不明点

勝手にMVP範囲を広げないでください。
決済、GPS常時追跡、チャット、写真管理、請求管理はPhase 2以降として扱ってください。
```

## 3. 技術構成を指定する場合の指示

Supabaseを使うなら、次の指示を追加します。

```text
技術構成は以下で進めてください。

- Next.js
- TypeScript
- React
- Supabase Auth
- Supabase PostgreSQL
- Supabase Storage
- Tailwind CSS

まずはMVPとして、ログイン、会社管理、現場管理、ダンプ管理、空き予定、マッチング、予約申請、予約承認、通知、操作履歴を作ります。
```

まずは以下を実装してください。

## 6. 予約機能を作る時の指示

予約は一番重要なので、単独で指示します。

```text
予約機能を実装してください。

必ず以下のルールを守ってください。

- 会社が承認済みでない場合は予約不可
- 現場状態がopenでない場合は予約不可
- 残り台数が0の場合は予約不可
- 同じ車両を同じ日程で二重予約できない
- 現場の必要スキルと車両スキルが一致する場合のみ申請可能
- 予約承認後は現場の残り台数に反映する
- 予約状態変更はreservation_status_logsに保存する
- 重要操作はaudit_logsに保存する

予約作成・承認はトランザクションで安全に処理してください。
```

## 7. Cursorに頼む時の注意

避けた方がよい指示:

```text
全部作って
いい感じに本番化して
資料を見て完成させて
```

おすすめの指示:

```text
まず資料を読んで、実装計画だけ出してください。
次にDBだけ作ってください。
次にログインだけ作ってください。
次に現場登録だけ作ってください。
次に予約承認だけ作ってください。
```

一気に作らせるより、区切って確認する方が安全です。

## 8. 開発開始前に人間が決めること

Cursorに実装させる前に、最低限以下を決めてください。

| 項目 | 決める内容 |
|---|---|
| 運営者の立場 | 紹介のみか、受発注管理まで行うか |
| 予約承認者 | 管理者のみか、現場会社も承認できるか |
| 手数料 | 現場会社、ダンプ会社、月額のどれか |
| ドライバー情報 | 電話番号を誰に表示するか |
| GPS | MVPでは使わないか、当日のみ使うか |
| 請求 | 最初は手動請求か、システム化するか |

未確定なら、MVPでは以下を仮ルールにします。

- 運営者は「紹介・予約管理」まで
- 予約承認は管理者のみ
- 請求・支払はMVPでは手動
- GPS常時追跡はMVPでは入れない
- ドライバー電話番号は予約確定後のみ表示

## 9. Cursorへ渡す短いまとめ文

短く依頼する場合は、以下で十分です。

```text
添付資料をもとに、ダンプ・現場マッチングアプリのMVP本番版を作りたいです。

最優先資料は「要件定義書_MVP.md」です。
DBは「DB設計書_MVP.md」に合わせてください。
「本番化設計メモ.md」は業務ルールの補足として使ってください。
「docs/prototype」はUI参考です。

まずコードを書かずに、実装方針・技術構成・作業順・不明点を出してください。
```
