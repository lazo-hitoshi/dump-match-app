# ダンプ・現場マッチングアプリ 本番化設計メモ

作成日: 2026-06-24  
対象: ダンプ会社、現場会社、管理者が使うマッチング・予約管理システム

## 1. Cursorエディタ等へ移すべきタイミング

今すぐ必ずCursorへ移す必要はありません。現在の段階では、試作画面と業務ルールを固めることが優先です。

ただし、本番開発に入る段階では、Cursor、VS Code、GitHub Codespacesなど、Git管理できる開発環境へ移すのが良いです。

| 段階 | おすすめ環境 | 理由 |
|---|---|---|
| 企画・試作 | 今のCodex環境で十分 | 画面案、業務ルール、データ項目を素早く作れる |
| MVP開発 | CursorまたはVS Code + GitHub | 複数ファイル、履歴管理、レビュー、修正がしやすい |
| 本番運用 | GitHub + CI/CD + 本番サーバー | 自動テスト、デプロイ、障害対応、権限管理が必要 |

Cursorは「開発を楽にするエディタ」です。データベース、ログイン、サーバー、決済、地図APIを代わりに用意してくれるものではないため、移行と同時に本番構成を決める必要があります。

推奨構成の例:

| 項目 | 候補 |
|---|---|
| フロントエンド | Next.js / React / TypeScript |
| データベース | PostgreSQL / Supabase |
| 認証 | Supabase Auth / Firebase Auth / Auth0 |
| 地図 | Google Maps Platform |
| ファイル保存 | Supabase Storage / S3 |
| 通知 | メール、LINE、SMS、Webプッシュ |
| 決済・請求 | 初期は請求書発行、将来はStripe等 |
| ホスティング | Vercel / AWS / Azure / Supabase |

## 4. 予約ルール表

### 4.1 予約ステータス

| 状態 | 意味 | 残り台数への影響 | 操作できる人 |
|---|---|---:|---|
| 下書き | 現場または予約が未公開 | 影響なし | 作成者、管理者 |
| 募集中 | 現場が応募受付中 | 影響なし | 現場会社、管理者 |
| 申請中 | ダンプ会社が応募した状態 | 原則影響なし | ダンプ会社、管理者 |
| 仮押さえ | 管理者が短時間確保した状態 | 任意で仮減算 | 管理者 |
| 承認済 | 予約が承認された状態 | 減算する | 管理者、現場会社 |
| 予約済 | 配車が確定した状態 | 減算する | 管理者 |
| 稼働中 | 当日作業中 | 減算する | 管理者、現場会社、ダンプ会社 |
| 完了 | 作業完了 | 減算対象外、実績対象 | 管理者 |
| キャンセル | 予約取消 | 減算しない | 管理者、条件により会社 |
| 却下 | 申請不採用 | 減算しない | 管理者、現場会社 |

MVPでは、残り台数を減らす状態は「承認済」「予約済」「稼働中」に限定するのが分かりやすいです。

### 4.2 予約可能条件

| ルール | 内容 | MVPで必須 |
|---|---|---|
| 会社承認 | 現場会社・ダンプ会社が承認済みである | 必須 |
| 車両有効 | 車両が停止状態ではない | 必須 |
| ドライバー有効 | ドライバーが停止状態ではない | 必須 |
| 日程一致 | 車両の空き期間が現場の予約期間を含む | 必須 |
| 重複禁止 | 同じ車両に同日同時間帯の予約がない | 必須 |
| 残台数確認 | 現場の残り台数が1台以上ある | 必須 |
| スキル一致 | 現場の必要スキルと車両・ドライバーの対応スキルが一致する | 必須 |
| 金額確認 | 現場単価と希望単価の差を表示する | 推奨 |
| 書類期限 | 車検、保険、免許などの期限切れを警告する | 推奨 |
| 管理者承認 | 予約確定前に管理者が確認する | MVPでは必須推奨 |

### 4.3 予約確定フロー

| 手順 | 処理 | 結果 |
|---:|---|---|
| 1 | 現場会社が現場を登録する | 現場状態は「募集中」 |
| 2 | ダンプ会社が空き予定を登録する | マッチ候補に出る |
| 3 | システムが日程・スキル・距離・金額で候補を表示する | おすすめ順に並ぶ |
| 4 | ダンプ会社が予約申請する | 予約状態は「申請中」 |
| 5 | 管理者または現場会社が確認する | 承認または却下 |
| 6 | 承認すると予約状態が「予約済」になる | 現場の残り台数が1減る |
| 7 | 当日、必要なら入場・退場を記録する | 実績データになる |
| 8 | 完了後、請求・支払へ進む | 経理処理対象になる |

### 4.4 台数計算ルール

| 項目 | 計算方法 |
|---|---|
| 予約台数 | 対象現場に紐づく「承認済」「予約済」「稼働中」の予約数 |
| 残り台数 | 必要台数 - 予約台数 |
| 充足判定 | 残り台数が0以下 |
| 募集中判定 | 現場状態が募集中、かつ残り台数が1以上 |
| 台数変更 | 必要台数を予約台数未満にはできない。管理者だけ例外処理可能 |

### 4.5 キャンセルルール

| 条件 | ルール |
|---|---|
| 予約確定前 | ダンプ会社または管理者が取消可能 |
| 予約確定後 | 原則、管理者承認が必要 |
| 前日・当日キャンセル | 理由入力を必須にする |
| 車両故障 | 代替車両の提案フローを出す |
| キャンセル料 | MVPでは手動管理。将来、自動計算 |
| キャンセル後 | 残り台数を再計算し、現場を募集中に戻す |

### 4.6 変更ルール

| 変更内容 | ルール |
|---|---|
| 現場日程変更 | 予約済み車両へ通知し、再承認が必要 |
| 必要台数変更 | 予約済み台数未満への変更は管理者のみ |
| 金額変更 | 予約済み分には反映しない。新規予約から適用 |
| 車両変更 | 同じ会社内の代替車両なら管理者承認で変更可能 |
| ドライバー変更 | 同じ会社内なら可能。ただし免許・スキル条件を再確認 |

### 4.7 マッチング優先順位

| 優先 | 条件 | 説明 |
|---:|---|---|
| 1 | 日程が合う | 空き期間が現場期間を満たす |
| 2 | スキルが合う | 大型、小型、深ダンプ、夜間など |
| 3 | 距離が近い | 待機場所または現在地から現場まで |
| 4 | 金額が合う | 希望単価が現場単価以下、または差が小さい |
| 5 | 書類が有効 | 期限切れ車両を下げる |
| 6 | 過去実績が良い | 将来拡張。キャンセル率、評価など |

| 操作履歴確認 | 可 | 自社分のみ | 自社分のみ | 不可 |

## 5. MVPで最初に作る範囲

最初の本番版は、機能を広げすぎず、次の範囲に絞るのが現実的です。

| 分類 | MVPに入れる |
|---|---|
| 認証 | ログイン、会社別権限、管理者権限 |
| 会社管理 | 会社登録、承認、停止 |
| 現場管理 | 登録、編集、一覧、詳細、残台数計算 |
| ダンプ管理 | 車両登録、空き予定、対応スキル |
| マッチング | 日程、スキル、距離、金額による候補表示 |
| 予約 | 申請、承認、予約済み、キャンセル |
| 通知 | メールまたはLINEで最低限の通知 |
| 地図 | 住所からGoogle Mapsを開く |
| 履歴 | 予約状態と重要変更の履歴 |

後回しでよいもの:

| 機能 | 理由 |
|---|---|
| 自動決済 | 契約・手数料設計が固まってから |
| GPS常時追跡 | 個人情報・同意・保存期間の設計が必要 |
| チャット | 予約管理が安定してから |
| 写真・QR入退場 | 実績管理フェーズで追加 |
| 自動評価 | 初期は手動メモで十分 |

## 7. 次に決めるべき質問

1. アプリ運営者は、予約の当事者になりますか。それとも紹介だけですか。
2. 予約承認は、管理者だけが行いますか。現場会社も承認できますか。
3. ダンプ会社から手数料を取りますか。現場会社から取りますか。
4. ドライバーの電話番号や現在地を、現場会社に表示しますか。
5. キャンセル料を自動計算しますか。最初は手動運用にしますか。
6. 支払いは、当面は請求書運用にしますか。アプリ内決済まで入れますか。
